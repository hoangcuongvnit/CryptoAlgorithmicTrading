## Migration Plan: Change Trading Sessions from 6x4h to 3x8h

### Goal
Move from 6 sessions/day (4 hours each) to 3 sessions/day (8 hours each), while keeping per-session phase logic unchanged:
- SoftUnwind window
- LiquidationOnly window
- ForcedFlatten window
- End-of-session full position close requirement

Target daily session windows (UTC):
- S1: 00:00 -> 08:00
- S2: 08:00 -> 16:00
- S3: 16:00 -> 24:00

---

## 2) Data and Backward Compatibility Risks

### Risk 1: Mixed historical data semantics
Existing `orders.session_id` values likely follow old model (`yyyyMMdd-S1..S6`).

If runtime changes to 8-hour sessions without migration strategy:
- New data uses S1..S3 semantics.
- Old data remains S1..S6 semantics.
- Range reports and session equity curves can become ambiguous or misleading.

### Risk 2: Parser/validator change can orphan old IDs
If `TryParseSessionId` is tightened directly to 1..3:
- Historical requests with S4..S6 will fail parsing.

### Risk 3: Frontend/backend deployment mismatch
If frontend is updated first:
- UI shows 3 sessions but backend may still return/report 6 buckets.

If backend is updated first:
- API semantics change before UI labels update.

---

## 3) Recommended Migration Strategy

### Decision principles
1. Runtime session logic must switch atomically to 8-hour windows.
2. Report semantics must be explicit for historical data.
3. Avoid silent remapping that can distort PnL analysis.

### Recommended approach
Use a dual-mode reporting strategy during transition:
- Mode `legacy-4h` for historical period (existing S1..S6 data).
- Mode `session-config` (new default) for all new dates/data.

Alternative (simpler but destructive):
- Recompute historical `session_id` into 8-hour buckets in DB migration.
- This makes history comparable in one model, but changes past labels and may break audit expectations.
