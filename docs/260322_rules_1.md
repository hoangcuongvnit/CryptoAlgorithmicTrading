# Trading Decision Rules (As-Built)

## 1. End-to-End Decision Flow

Current runtime flow is:

1. Ingestor publishes price ticks to Redis.
2. Analyzer computes indicators and emits `TradeSignal`.
3. Strategy maps `TradeSignal` to `OrderRequest`.
4. RiskGuard validates or adjusts the order.
5. Executor places live order and persists the result.
6. Session services can force reduce-only liquidation near session end.
7. Capital ledger tracks capital allocation and session snapshots.

This means buy/sell decisions are not made by one rule only. They are a layered pipeline:

- Signal quality selection (Analyzer strength + Strategy minimum strength)
- Direction and pricing mapping (Strategy)
- Risk filters and caps (RiskGuard)
- Final execution gates (Executor state/session/shutdown)

## 2. Buy/Sell Decision Criteria

## 2.1 Indicator and Signal Strength (Analyzer)

Analyzer calculates:

- RSI (default period 14)
- EMA short (default 9)
- EMA long (default 21)
- Bollinger Bands (period 20, stddev 2.0)

Signal strength is evaluated as:

- Bullish context: `EMA9 >= EMA21`
- Bearish context: `EMA9 < EMA21`

Confirmations:

- Bullish confirmations:
	- `RSI < RsiOversoldThreshold` (default 35)
	- `Price <= BB_Lower * 1.005`
- Bearish confirmations:
	- `RSI > RsiOverboughtThreshold` (default 65)
	- `Price >= BB_Upper * 0.995`

Strength output:

- 2 confirmations -> `Strong`
- 1 confirmation -> `Moderate`
- 0 confirmation -> `Weak`

## 2.2 Strategy Mapping to Buy/Sell

Strategy does not directly consume a side from Analyzer; it derives side from EMA relation:

- `Buy` when `signal.Ema9 >= signal.Ema21`
- `Sell` when `signal.Ema9 < signal.Ema21`

Entry price:

- `entry = signal.BbMiddle` if > 0
- else fallback `entry = signal.Ema9`

Signal filter:

- Reject if signal strength is below `Trading.MinimumSignalStrength` (default `Moderate`).
- In `SoftUnwind` phase, only `Strong` signals are accepted.

Order risk targets generated by strategy:

- Buy:
	- `StopLoss = entry * 0.985` (-1.5%)
	- `TakeProfit = entry * 1.03` (+3.0%)
- Sell:
	- `StopLoss = entry * 1.015` (+1.5% adverse)
	- `TakeProfit = entry * 0.97` (-3.0% favorable)

Default strategy quantity:

- `Trading.DefaultOrderQuantity` (default `0.001`).

## 3. Risk Validation Rules (Order of Execution)

RiskGuard evaluates rules in this exact order, short-circuiting on first rejection:

1. `RecoveryWindowRule`
2. `NoCrossSessionCarryRule`
3. `SessionWindowRule`
4. `SymbolAllowListRule`
5. `QuantityRule`
6. `PositionSizeRule` (can adjust quantity down, not necessarily reject)
7. `RiskRewardRule`
8. `CooldownRule`
9. `MaxDrawdownRule`

### Key risk criteria

- Recovery gate: blocks new non-reduce-only orders during startup recovery states.
- Session gate: new positions allowed only in `Open` phase.
- Cross-session protection: order session must match current session.
- Symbol allowlist: only configured symbols are tradable when list is non-empty.
- Quantity sanity: quantity and entry price must be > 0.
- Position sizing:
	- Max notional per order is computed from percentage cap when configured:
		`MaxNotional = MaxPositionSizePercent/100 * VirtualAccountBalance`
	- If order exceeds cap, quantity is reduced to fit.
- Risk/reward minimum:
	- Must satisfy configured `MinRiskReward` (default 2.0).
- Cooldown:
	- Minimum time gap per symbol between approved orders (default 30s).
- Max drawdown:
	- Uses today's net realized PnL from closed orders.
	- Rejects all new orders if loss breaches configured daily limit.

## 4. Trade Types and Execution Behavior

## 4.1 Supported Order Types

System supports these order types in shared contracts:

- `Market`
- `Limit`
- `StopLimit`

## 4.2 Types Used by Current Runtime Paths

- Strategy-generated normal trades: `Market`
- Liquidation orchestrator:
	- `Limit` in `LiquidationOnly` phase
	- `Market` in `ForcedFlatten` phase or emergency flatten
- Shutdown close-all: `Market` reduce-only

## 4.3 Side Semantics in Position Tracking

- `Buy` increases/open positions.
- `Sell` reduces/closes positions.
- Realized PnL is recognized when sell fills against tracked long inventory.

Current in-memory tracker is long-oriented (buy-open, sell-close). There is no separate short inventory model implemented in position tracker.

## 5. Cash Allocation and Capital Distribution

Cash allocation is governed primarily by risk caps and session constraints.

## 5.1 Per-Order Capital Allocation

Primary cap is RiskGuard `PositionSizeRule`:

- `MaxPositionSizePercent` (default 2.0%) of `VirtualAccountBalance` (default 10,000)
- Effective max notional per order default:
	- `2% * 10,000 = 200` USDT

If requested order notional is above this cap, RiskGuard reduces quantity to the maximum allowed size.

## 5.2 Additional Allocation Constraints

- RiskReward filter removes low-quality risk profiles.
- Cooldown prevents rapid repeated allocation on same symbol.
- MaxDrawdown globally halts new risk after daily loss limit breach.
- Executor has an additional `Trading.MaxNotionalPerOrder` gate (default 1000).

Operationally, the stricter layer wins. With current defaults, RiskGuard percentage cap is usually stricter than Executor absolute cap.

## 5.3 Session-Based Allocation Behavior

- Session phases:
	- `Open` -> new positions allowed
	- `SoftUnwind` -> strategy only accepts `Strong` signals
	- `LiquidationOnly` -> only closing/reduce-only activity
	- `ForcedFlatten` -> aggressive flattening before session end
- Near end of session, system shifts from alpha-seeking to risk-off capital protection.

## 5.4 Capital Ledger

Capital tracking in the system:

- Budget status (cash, ROI, realized PnL)
- Ledger events (`DEPOSIT`, `WITHDRAW`, `RESET`, `SESSION_PNL`)
- Session open/close capital snapshots
- Equity curve and capital flow timeline APIs

This ledger records cash evolution and session performance. Per-order allocation decisions still come from risk/execution gates above.

## 6. Current Default Configuration Snapshot

Analyzer defaults:

- RSI period: 14
- EMA short/long: 9/21
- BB period/stddev: 20 / 2.0
- Oversold/Overbought: 35 / 65
- Signal interval: 1m candle

Strategy defaults:

- Minimum signal strength: `Moderate`
- Default quantity: `0.001`

RiskGuard defaults:

- Min risk/reward: `2.0`
- Max position size: `2.0%`
- Virtual account balance: `10,000`
- Max drawdown: `5.0%` per day
- Cooldown: `30s`
- Allowed symbols: BTCUSDT, ETHUSDT, BNBUSDT, SOLUSDT, XRPUSDT

Executor defaults:

- Global kill switch: off
- Max notional per order: `1000`
- Binance API retry: 5 retries with exponential backoff

Session defaults:

- Session length: 4 hours
- Soft unwind: 15 minutes before liquidation window
- Liquidation window: last 30 minutes of session
- Forced flatten: last 10 minutes

## 7. Important Notes and Gaps

1. `SessionSettings.MaxOpenPositionsPerSession` exists in config but is not currently enforced by a dedicated active rule in the runtime path.
2. `SessionOpenCooldownSeconds` exists in settings but is not currently applied by core trading policy logic.
3. Strategy direction is derived from EMA relation only; RSI/BB affect signal strength, not side selection directly.
4. RiskGuard cooldown state is persisted to Redis for continuity across restarts.

## 8. Practical Interpretation

If you ask, "When does the system buy or sell?" the current answer is:

- Buy when EMA9 is above/equal EMA21, signal strength passes threshold, session allows new entries, and all risk rules pass.
- Sell when EMA9 is below EMA21 under the same gates.
- Forced sell/reduce-only can also occur from liquidation and shutdown workflows, independent of new alpha signal generation.

If you ask, "How is money allocated?" the current answer is:

- Base quantity starts from strategy default.
- RiskGuard caps it to max notional based on percent-of-virtual-balance.
- Additional filters (RR, cooldown, drawdown, allowlist, session/recovery state) can block the trade entirely.
